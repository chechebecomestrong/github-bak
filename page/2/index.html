ui自动化测试探索——ui recorder

发表于 2018-09-04 | | 阅读次数:
字数统计: 1,209 | 阅读时长 ≈ 4

最近在写npm包相关的内容。
在写到单元测试相关内容的时候，想到当前的单元测试只限于检查一系列的js代码的正确性和相关性。
于是，想了下能不能对UI进行自动化测试？
答案是肯定的。
首当其冲的，则是UI Recorder。
是阿里巴巴的开源项目。
官方的介绍十分的高大上：

UI Recorder 是一款零成本UI自动化录制工具，类似于Selenium IDE.
UI Recorder 要比Selenium IDE更加强大!
UI Recorder 非常简单易用.

官方网站: http://uirecorder.com/
语言切换: English, 中文
变更日志: CHANGE
视频教程：PC中文教程
功能
支持所有用户行为: 键盘事件, 鼠标事件, alert, 文件上传, 拖放, svg, shadow dom
支持无线native app(Android, iOS)录制, 基于macaca实现: https://macacajs.com/
无干扰录制: 和正常测试无任何区别，无需任何交互
录制用例存储在本地
支持丰富的断言类型: val,text,displayed,enabled,selected,attr,css,url,title,cookie,localStorage,sessionStorage
支持图片对比
支持强大的变量字符串
支持公共测试用例: 允许用例中动态调用另外一个
支持并发测试
支持多国语言: 英文, 简体中文, 繁体中文
支持单步截图
支持HTML报告和JUnit报告
全系统支持: Windows, Mac, Linux
基于Nodejs的测试用例: jWebDriver
因为node操作起来相对较为简便。所以选择用node来探索。
工欲善其事必先利其器
1、安装
首先安装node 安装Chrome 安装ui recorder、mocha
安装完之后，需要对环境进行补充，需要另外安装Java sdk，jwedriver
在把环境都安装好之后，就可以进行测试了。
2、测试步骤
首先，新建一个文件夹。
然后进入测试环节：
（1）uirdcorder init 配置好之后在用ui recorder的环境 会生成一些配置文件
（2）生成的json文件中，可以修改config.json使配置更加符合自己的情况，而且也可以修改在init的时候配置错误的选项。
（3）修改好配置文件后，就可以开始录制测试用例了。
指令为 uirecorder start 测试用例名
测试用例名以.spec.js结尾
然后会有一些关于录制时的配置需要进一步的确定。都确认之后，终端会成为所谓的录制服务器，内部的驱动会拉起系统的浏览器。
这个时候会拉起两个浏览器，一个是录制浏览器，一个是检验浏览器，所有的录制动作都是在录制浏览器进行的，检验浏览器会重复录制浏览器的动作，然后在录制服务器中显示测试结果。
录制浏览器会有一个输入框，即输入要测试的内容，这里可以是一个url，也可以是一个脚本。点击开始录制即真正了进入了测试过程。
录制界面会有一排按钮框，悬浮可移动。
有添加悬停、添加断言、使用变量、脚本跳转、结束录制等功能。
在经过一些列的测试之后，可以结束录制。这个时候在录制中的所有的操作都以代码的形式出现在刚刚的.spec.js文件中。
所以，这个文件可以用来回看。回看需要mocha。
mocha 测试用例名
这样就可以回看刚刚的录制。
然后在终端，会出现各种测试动作的结果。
有mocha的加入，就有了可以文本化测试结果的可能。
source run.sh 测试用例名
这个时候除了会回放录制的动作，回放完后会生成一份测试报告
测试报告中除了一般的单元测试测试报告中显示的各项数据外，还有录制动作的截图。
可以实时、带代码块、带测试结果的了解整个测试过程。
金无足赤人无完人。
软件也都不是完美的：
这款测试框架：
只适用于一般的dom操作，以及一般的dom写的ui。
不适合我们公司的ui框架。因为我们公司的ui框架写的很精细，正常的写法测不出来。
如果硬写也能写出来，但是到时候测试的代码可能比开发代码还多。这样就会导致测试的成本过高。
因此，这个ui测试，以后可以在表格、一般的dom操作中使用。不适合用来测试本公司前端的ui框架。
所以，这个ui测试框架，不再做进一不的探究。需要的时候，用其基本的写法即可。

2018年8月6日
Git一般的使用场景：
一般是旧项目
一般直接git clone
修改完后再次上传的时候 配置文件也是之前写好的

然鹅……
今天下午，面对一个新的项目，什么都没有……然后，git就变得很佛性

分享一个佛性的我面对佛性的git的心路历程：
按部就班，
git init
git add .
git commit
然后之后git push就可以了
但是，失败了。
发现，是因为没有master。
没有master，什么都是0。
添加了master后，打了个分支，然后高高兴兴上传。
依旧失败。
查找原因，可能是本地分支没有关联远程分支的原因。
删除分支，重新创建分支。
上传成功！……但是发现：文件结构多了一层。
删除分支，重新上传。
上传成功！……但是发现：有node_modules和mac自带的文件
于是想起来要有一个gitignore文件。
写了个最简单的gitignore文件：

删掉分支，再次上传。
gitignore并没有发挥作用。
重写了gitignore但是没有用。
然后，考虑到git的缓存的问题。
于是
git status
真的是有缓存文件。
git clean出场
删除 untracked files
git clean -f

连 untracked 的目录也一起删掉
git clean -fd

连 gitignore 的untrack 文件/目录也一起删掉 （慎用，一般这个是用来删掉编译出来的 .o之类的文件用的）
git clean -xfd
以上这三条，没有起作用。
最后尝试，
git rm -r --cached .
这才终于把缓存清掉。
上传。成功！git ignore也发挥了作用。
git虽然几乎天天用，但是经常用于旧项目，在应对新项目时竟然出了这么多的幺蛾子、
基础是进步的基石，共勉！

code review 清单

发表于 2018-07-30 | | 阅读次数:
字数统计: 2,157 | 阅读时长 ≈ 7
  • 把使用清单作为起点，不断根据需求的变化进行优化。
  • 多多记录在代码审查时发现的问题，就能够确定那些常犯的错误，然后量身定制一个清单。
  • 删除那些没有出现过的错误
  • 也可以保留那些出现概率很小，但是非常关键的项目，比如安全相关的问题

下面是整理的在code的时候可能出现的问题：
常见代码问题
空值
防御式编程，判空处理，有胜于无
未捕获潜在的异常
API返回做判断
切忌默默吞掉异常（线上的有些代码可能会捕捉或者抛出异常，但是在code review的时候要尽量弄懂异常出现的原因以及探索解决的方法）
防御式编程
低性能
批量调用接口
处理超时
降维多重循环
多层对象嵌套调用
影响范围过大
谨慎修改公共函数、基础函数、基类等
充分测试
评估影响范围
单元测试问题
重要业务逻辑必有单元测试
单元测试需要全部通过
代码BUG或变更单元测试覆盖
增加异常单元测试
与原有业务逻辑不兼容
充分理解已有逻辑及设计
缺乏必要日志
重要而关键的实例状态、代码路径及API调用，INFO日志以及预料外传入的参数
异常捕获Error
错误码不符合规范
遵循团队规范
参数检测缺乏或不足
参数值类型
参数业务类型
契约式编程
API参数传输错误
参数类型具体化
避免相邻的参数是同类型
引用错误
有多个引用时应该检查引用名称，避免在跨度比较大的范围内使用多个引用
添加新的方法名时避免重名
细节错误
可选布尔参数默认值应为false
数组越界
JSON解析出错
函数参数传递出错
API版本不对
使用未经测试的代码
不成熟的算法
相等比较
避免依赖类型转换
多重条件
拆分为有业务语义的单个条件的组合
文不符实
查询接口中有更新/新增甚至删除操作，引发故障的定时炸弹
跨语言交互
空值、json、数组的不同解析，encodeURI类的方法行为的不一致
接口与交互约定
开发接口设计要符合小而美，正交的特性
命名要贴切一致
参数要指明约束
枚举参数要给出列表
结果返回要规范一致，尽可能的给出返回值，如无必要，避免写void方法
联调确认
可维护性问题
可维护问题是在当前业务变更的范围内通常不会导致bug、故障，却会在日后埋雷，引发bug和故障、维护成本大幅增加
硬编码
多处定义某个变量，并且变量的值相同：使用含义明显的枚举或常量
写死的配置（配置项）：不影响业务功能，但出现环境变更的时候就会稍显麻烦。尽量将配置抽离出来做成配置项放到配置文件里
临时加的逻辑和文案（抽离管理）：抽离出来放在单独的函数或方法里，并特别加以注释。
重复代码
维护事实一致性：抽离通用的部分，定制差异
创造新函数时，先看看是否有此类方法已经实现过
通用逻辑与业务逻辑耦合
先祈福，然后抽离一个新的函数，严格判断相应条件满足后去调用
如果是新创建逻辑，使用函数式编程或基于接口的编程：可以将通用处理流程抽离出来，而将具体业务逻辑以回调函数的形式传入处理
不要让不同的业务共用相同的函数，每个业务都有各自的函数，并可复用相同的通用逻辑和流程处理
通用逻辑应该确保功能单一
复用而非混杂
直接在原方法里加逻辑
增加了原方法的长度，破坏了可维护性，另一方面，对原方法的既有逻辑造成破坏。
方法：新增一个函数，然后在原方法中调用并说明原因（或者在条件允许的时候反过来，新方法调用原方法）
多业务耦合
先划分业务边界，定义好接口设计和服务依赖关系，再着手开发
代码层次不合理
代码改动逻辑是正确的，然而代码的存放位置不符合当前架构设计约定，导致后续维护成本增加
代码的放置位置不符合当前架构设计约定
代码层次不合理可能导致重复代码
多余或废弃的代码
工程中有一些不用的代码或者是暂时用到的util工具或库函数，或者由于业务变更导致已经废弃不用的代码，或者后来又重写的代码。
后续维护时会造成困惑
方法：确认不用及时删除
使用全局变量
全局变量并没有“错”，但是一旦出现问题，排查和调试问题的时候会比较麻烦
破坏函数的复用性，导致可维护性变差。
代码块越长，全局变量的作用域越广，不易理解和调试
缺乏必要的解释
重要和关键点的代码应该有必要的解释，使用的重要算法缺乏必要的引用出处。对特殊的处理要有必要的说明
原则上，每个方法至少用一个简短的单行注释，适宜地描述方法的用途、业务逻辑、作者及日期。
对于特殊甚至奇葩的需求的特别实现，更要加一些注释，从而后续维护有基础。
较轻微问题
没有技术难度，通过良好的习惯即可避免的问题
命名不贴切
取贴切的名字
声明时未初始化
赋予默认初始值
风格与整体不一致
在工程项目中，遵循工程约定的风格
在个人项目中，可以凸显个性风格
类型转换错误
转换前判断类型
避免类型转换
否定式风格
使用肯定句式，避免双重否定
API参数传递错误
留意顺序正确
参数太多时，使用key-value对象作为options或config传递
单行调用括号过多
在写一个语句内有多个参数多个括号的时候，如果再结合api参数传递错误，IDE没有报错，并且自己也没有意识到错误。
可以拆除一部分变量，并将调用之间的括号用空格隔开，显示出层次感
修改方法签名
不要修改原方法的参数顺序
在最后面增加可选参数
最好的方法是：新建一个方法取复用原方法，然后调用新方法。
打印日志太多
仅保留必需日志
多级数据结构
确保父级结构的数据存在
作用域过大
仅在需要时定义和使用
分支与循环
if-else嵌套条件，需要仔细检查是否符合业务逻辑，如果嵌套太深，是否可以换另一种方式解结
switch语句，case要注意加break，最好加default处理
for循环，要注意结束条件合理避免死循环。对于循环变量的控制，避免出现+1或-1的错误，消除越界错误
注*for循环也要特别注意空值和空容器的处理，避免抛出空值异常。通过单测来确保for循环的准确性
复杂隐蔽问题
复杂并发场景下的有一定技术难度的、需要丰富开发与设计经验才能看出来的错误
安全问题
严格检查和屏蔽非法输入
敏感信息加密通信
业务处理后消除敏感私密问题
角色、权限、操作、数据访问分级和控制
设计问题
潜在的性能问题

扩展性问题
其他可能遗漏的问题

随着写的代码量不断的增加，慢慢就接触了封装这个概念。

当时是web的上机课：

当是是学了用orm来映射数据库后，顺理成章的也写进了框架里。
老师给我们定了3个任务。
1、查询所有的用户

node中很容易出现异步的问题，为了避免出现钩子陷阱，所以出现了promise这种写法。
从而就会有这样的情景：
在某个方法中，promisify的request连同获取的数据也一并封在了promise对象里。
要想把promise中的数据取出来，我也想了很多办法，像处理平时的对象那样，以把对象转化成字符这种思想为中心，各种方法用过了一遍。
好吧，最后承认promise对象和其他的对象不一样。
只能去网络上搜索相关的方。
首先是这么一种答案：取不出来的。
然后呢？然后就没有然后了。
继续搜，最后终于有了另一种答案：promise里取不出，可以换个方法啊，比如用async/await之类的，反正和promise一样都是消除异步的……这个答案，相当于是曲线救国了。换async/await是可以，但是如果换，就要整个封装的函数进行大改，而且，async/await属于新语法，支不支持还不知道……再结合我要实现的功能，感觉前面还有一些个大坑等着我。
既然写成了promise而且也把request promisify了，硬着头皮，进一步搜索。
果然，搜索引擎是强大的。
找到了一个明白人说的答案，显然是可以把值取出来。
我理解的是这样的：
首先有这样的前提：
promise有三个状态，pending，resolve，reject。每个promise都是从pending到resolve或者reject。当promise到了resolve或者reject状态后，peomise就不再变了。
基于这样的前提下：
那么在promise中的数据也就不会变了。
当return这个promise后，promise的数据也会被return。
好的，下一步就是特别重要的.then()方法
也许，很多人一看，这有什么，我也用过，不就是promise.then()方法吗？
但是这个.then()方法值得我们好好讨论一下
——此处占位写.then()的相关资料——-
在看了很多的资料后，有一个特别通俗的说法：这个.then()方法可以认为是一个callback()，后面可以搭配.catch()可以捕捉来自promise里的错误，不用设置reject，就可以把错误捕捉到。

基础是进步的基石。

JS和HTML的交互基于事件。

事件流
描述的是从页面中接受事件的顺序
IE：事件冒泡流
Netscape：事件捕获流

事件冒泡：即事件最开始由最具体的元素（文档中嵌套层次最深的那个节点）接收，逐级向上传播至最不具体的那个节点（文档）。
click在input->div->body->document

事件捕获：不太具体的节点应该更早接收到事件， 而最具体的节点最后接收到事件。
document->body->div->input
使用事件处理程序
1、HTML事件处理程序
把事件直接加在HTML结构里。
缺点：HTML和JS代码紧密耦合
2、DOM0级事件处理程序
把一个函数赋值给一个事件的处理程序属性，是用的比较多的方法
简单，跨浏览器
先取一个元素，把事件作为元素的属性，没有HTML事件的缺点。
var btn = document.getElementById(‘dd);
btn.onclick=function(){}
btn.onclick=null;//取消了onclick事件
3、DOM2级事件处理程序
定义了两个方法：
用于处理指定和删除事件处理程序的操作
addEventListener()和removeEventListener()
接收3个参数：
要处理的事件名、作为事件处理程序的函数和布尔值（true：事件捕获，false：事件冒泡）
var btn = document.getElementById(‘dd);
btn.addEventListener(‘click’,fun,false); //onclick换成click，onmouseover换成mouseover,以此类比，设置成false作为一个冒泡事件从而让大多数浏览器兼容
btn.removeEventListener(‘click’,fun,false) //传递和addEventListenery一样的参数才能删除，而且addEventListener的事件必须由removeEventListener才能删除
DOM0级和DOM2级可以添加多个事件以及多个事件处理程序
但是IE不支持，因此以下是IE的事件处理程序
4、IE事件处理程序
两个参数，事件处理程序的名称和事件处理程序的函数
attachEvent()添加事件
detachEvent()删除事件
不使用第三个参数的原因是：IE8以及以前的版本只支持事件冒泡。
var btn = document.getElementById(‘dd);
btn.attachEvent(‘onclick’,fun); //这个时候的事件又要加on了，后面第二个参数是函数名
btn.detachEvent(‘onclick’,fun) //删除事件
5、跨浏览器的事件处理程序
可以封装在一个对象内，能力检测
var eventUtil = {
