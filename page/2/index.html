按部就班，
git init
git add .
git commit
然后之后git push就可以了
但是，失败了。
发现，是因为没有master。
没有master，什么都是0。
添加了master后，打了个分支，然后高高兴兴上传。
依旧失败。
查找原因，可能是本地分支没有关联远程分支的原因。
删除分支，重新创建分支。
上传成功！……但是发现：文件结构多了一层。
删除分支，重新上传。
上传成功！……但是发现：有node_modules和mac自带的文件
于是想起来要有一个gitignore文件。
写了个最简单的gitignore文件：

删掉分支，再次上传。
gitignore并没有发挥作用。
重写了gitignore但是没有用。
然后，考虑到git的缓存的问题。
于是
git status
真的是有缓存文件。
git clean出场
删除 untracked files
git clean -f

连 untracked 的目录也一起删掉
git clean -fd

连 gitignore 的untrack 文件/目录也一起删掉 （慎用，一般这个是用来删掉编译出来的 .o之类的文件用的）
git clean -xfd
以上这三条，没有起作用。

git rm -r --cached .
这才终于把缓存清掉。

硬编码
多处定义某个变量，并且变量的值相同：使用含义明显的枚举或常量
写死的配置（配置项）：不影响业务功能，但出现环境变更的时候就会稍显麻烦。尽量将配置抽离出来做成配置项放到配置文件里

维护事实一致性：抽离通用的部分，定制差异

通用逻辑与业务逻辑耦合
先祈福，然后抽离一个新的函数，严格判断相应条件满足后去调用
如果是新创建逻辑，使用函数式编程或基于接口的编程：可以将通用处理流程抽离出来，而将具体业务逻辑以回调函数的形式传入处理
不要让不同的业务共用相同的函数，每个业务都有各自的函数，并可复用相同的通用逻辑和流程处理
通用逻辑应该确保功能单一
复用而非混杂
直接在原方法里加逻辑
增加了原方法的长度，破坏了可维护性，另一方面，对原方法的既有逻辑造成破坏。
方法：新增一个函数，然后在原方法中调用并说明原因（或者在条件允许的时候反过来，新方法调用原方法）
多业务耦合
先划分业务边界，定义好接口设计和服务依赖关系，再着手开发
代码层次不合理
代码改动逻辑是正确的，然而代码的存放位置不符合当前架构设计约定，导致后续维护成本增加
代码的放置位置不符合当前架构设计约定
代码层次不合理可能导致重复代码
多余或废弃的代码
工程中有一些不用的代码或者是暂时用到的util工具或库函数，或者由于业务变更导致已经废弃不用的代码，或者后来又重写的代码。
后续维护时会造成困惑
方法：确认不用及时删除
使用全局变量
全局变量并没有“错”，但是一旦出现问题，排查和调试问题的时候会比较麻烦
破坏函数的复用性，导致可维护性变差。
代码块越长，全局变量的作用域越广，不易理解和调试
缺乏必要的解释
重要和关键点的代码应该有必要的解释，使用的重要算法缺乏必要的引用出处。对特殊的处理要有必要的说明
原则上，每个方法至少用一个简短的单行注释，适宜地描述方法的用途、业务逻辑、作者及日期。
对于特殊甚至奇葩的需求的特别实现，更要加一些注释，从而后续维护有基础。
较轻微问题
没有技术难度，通过良好的习惯即可避免的问题
命名不贴切
取贴切的名字
声明时未初始化
赋予默认初始值
风格与整体不一致
在工程项目中，遵循工程约定的风格
在个人项目中，可以凸显个性风格
类型转换错误
转换前判断类型
避免类型转换
否定式风格
使用肯定句式，避免双重否定
API参数传递错误
留意顺序正确
参数太多时，使用key-value对象作为options或config传递
单行调用括号过多
在写一个语句内有多个参数多个括号的时候，如果再结合api参数传递错误，IDE没有报错，并且自己也没有意识到错误。
可以拆除一部分变量，并将调用之间的括号用空格隔开，显示出层次感
修改方法签名
不要修改原方法的参数顺序
在最后面增加可选参数
最好的方法是：新建一个方法取复用原方法，然后调用新方法。
打印日志太多
仅保留必需日志
多级数据结构
确保父级结构的数据存在
作用域过大
仅在需要时定义和使用
分支与循环
if-else嵌套条件，需要仔细检查是否符合业务逻辑，如果嵌套太深，是否可以换另一种方式解结
switch语句，case要注意加break，最好加default处理
for循环，要注意结束条件合理避免死循环。对于循环变量的控制，避免出现+1或-1的错误，消除越界错误
注*for循环也要特别注意空值和空容器的处理，避免抛出空值异常。通过单测来确保for循环的准确性
复杂隐蔽问题
复杂并发场景下的有一定技术难度的、需要丰富开发与设计经验才能看出来的错误
安全问题
严格检查和屏蔽非法输入
敏感信息加密通信
业务处理后消除敏感私密问题
角色、权限、操作、数据访问分级和控制
设计问题

从而就会有这样的情景：
在某个方法中，promisify的request连同获取的数据也一并封在了promise对象里。
要想把promise中的数据取出来，我也想了很多办法，像处理平时的对象那样，以把对象转化成字符这种思想为中心，各种方法用过了一遍。
好吧，最后承认promise对象和其他的对象不一样。
只能去网络上搜索相关的方。
首先是这么一种答案：取不出来的。
然后呢？然后就没有然后了。
继续搜，最后终于有了另一种答案：promise里取不出，可以换个方法啊，比如用async/await之类的，反正和promise一样都是消除异步的……这个答案，相当于是曲线救国了。换async/await是可以，但是如果换，就要整个封装的函数进行大改，而且，async/await属于新语法，支不支持还不知道……再结合我要实现的功能，感觉前面还有一些个大坑等着我。
既然写成了promise而且也把request promisify了，硬着头皮，进一步搜索。
果然，搜索引擎是强大的。
找到了一个明白人说的答案，显然是可以把值取出来。
我理解的是这样的：
首先有这样的前提：
promise有三个状态，pending，resolve，reject。每个promise都是从pending到resolve或者reject。当promise到了resolve或者reject状态后，peomise就不再变了。
基于这样的前提下：
那么在promise中的数据也就不会变了。
当return这个promise后，promise的数据也会被return。
好的，下一步就是特别重要的.then()方法
也许，很多人一看，这有什么，我也用过，不就是promise.then()方法吗？
但是这个.then()方法值得我们好好讨论一下
——此处占位写.then()的相关资料——-
在看了很多的资料后，有一个特别通俗的说法：这个.then()方法可以认为是一个callback()，后面可以搭配.catch()可以捕捉来自promise里的错误，不用设置reject，就可以把错误捕捉到。

基础是进步的基石。

JS和HTML的交互基于事件。

事件流
描述的是从页面中接受事件的顺序
IE：事件冒泡流
Netscape：事件捕获流

事件冒泡：即事件最开始由最具体的元素（文档中嵌套层次最深的那个节点）接收，逐级向上传播至最不具体的那个节点（文档）。
click在input->div->body->document

事件捕获：不太具体的节点应该更早接收到事件， 而最具体的节点最后接收到事件。
document->body->div->input
使用事件处理程序

缺点：HTML和JS代码紧密耦合

简单，跨浏览器
先取一个元素，把事件作为元素的属性，没有HTML事件的缺点。

btn.onclick=function(){}
btn.onclick=null;//取消了onclick事件

display

display是CSS中最重要的用于控制布局的属性，每个元素都有一个默认的display值，这与元素的类型有关，大多数元素它们的默认值通常是block或者inline。一个block元素通常被叫做块级元素，一个inline元素通常被叫做行内元素。
block
div是一个标准的块级元素。一个块级元素会新开始一行并且尽可能撑满容器。其他常用的块级元素包括p、form和HTML5中的新元素：header、footer、section等等
inline
span是一个标准的行内元素。一个行内元素可以在段落中zz包裹一些文字而不会被打乱段落的布局。a元素是最常用的行内元素，它可以被用作链接。
none
另一个常用的display值是none。一些特殊元素的默认display值就是它。display：none通常被JavaScript用来在不删除元素的情况下隐藏活显示元素。它和visibility属性不一样。把display设置成none元素不会占据它本来应该显示的空间，但是设置成visibility：hidden，还会占据空间。
其他display值

其他
每个元素都有一个默认的display类型，不过可以随时随地的重写。

margin：auto

设置块级预算元素的width可以防止它从左到右撑满整个容器。然后你就可以设置左右外边距为auto来使其水平居中。元素会占据你所指定的宽度，然后剩余的宽度会一分为二成为左右外边距。
唯一的问题是，当浏览器窗口比元素的宽度还要宅时，浏览器会显示一个水平滚动条来容纳页面。然后就有了以下的解决方案。

max-width

在这种情况下使用max-width替代width可以使浏览器更好的处理小窗口的情况，在移动设备上显得尤为重要。
所有的主流浏览器包括IE7+在内都支持max-width，所以可以放心使用

盒模型

当你设置了元素的宽度，实际展现的元素却超出了你的设置：这是因为元素的边框和内边距会撑开元素。

position

  1. static
    static是默认值，任意的position：static的元素不会被特殊的定位。一个static元素表示它不会被“positioned”，一个position属性被设置为其他值的元素表示它会被“positioned”。
  2. relative
    relative表现的和static一样，除非你添加了一些额外的属性。在一个相对定位的元素上设置top、bottom、left、right属性会使其偏离其正常位置。其他的元素的位置则不会受该元素的影响发生位置改变来弥补它偏离后剩下的空隙。
  3. fixed
    一个固定定位的元素会相对于视窗来定位，这意味着即便页面滚动，它还是会停留在相同的位置，和relative一样，top、bottom、left、right属性都可用。
    一个固定定位的元素不会保留它原本在页面中应有的空隙，即脱离文档流。
    但是移动浏览器对fixed的支持很差，尤其是前期ios系统对webview进行的“优化”。对于移动端页面开发，这个属性需要对各种手机型号、各种手机系统以及客户使用的版本进行测试。必要的时候还是使用position：absolute，让这个元素回归文本流。
  4. absolute
    absolute是最棘手的position值。与fixed的表现类型，但是它不是相对于视窗而是相对于最近的“positioned”祖先元素。如果绝对定位的元素没有“positioned”祖先元素，那么它是相对于文档的body元素并且它会随着页面滚动而滚动。
