git佛系命令

发表于 2018-08-06
字数统计: 449 | 阅读时长 ≈ 2

2018年8月6日
Git一般的使用场景：
一般是旧项目
一般直接git clone
修改完后再次上传的时候 配置文件也是之前写好的

然鹅……
今天下午，面对一个新的项目，什么都没有……然后，git就变得很佛性

分享一个佛性的我面对佛性的git的心路历程：
按部就班，
git init
git add .
git commit
然后之后git push就可以了
但是，失败了。
发现，是因为没有master。
没有master，什么都是0。
添加了master后，打了个分支，然后高高兴兴上传。
依旧失败。
查找原因，可能是本地分支没有关联远程分支的原因。
删除分支，重新创建分支。
上传成功！……但是发现：文件结构多了一层。
删除分支，重新上传。
上传成功！……但是发现：有node_modules和mac自带的文件
于是想起来要有一个gitignore文件。
写了个最简单的gitignore文件：
node_modules/
node_modules/*
.DS_Store
删掉分支，再次上传。
gitignore并没有发挥作用。
重写了gitignore但是没有用。
然后，考虑到git的缓存的问题。
于是
git status
真的是有缓存文件。
git clean出场
删除 untracked files
git clean -f

连 untracked 的目录也一起删掉
git clean -fd

连 gitignore 的untrack 文件/目录也一起删掉 （慎用，一般这个是用来删掉编译出来的 .o之类的文件用的）
git clean -xfd
以上这三条，没有起作用。
最后尝试，
git rm -r –cached .
这才终于把缓存清掉。
上传。成功！gitignore也发挥了作用。
git虽然几乎天天用，但是经常用于旧项目，在应对新项目时竟然出了这么多的幺蛾子、
基础是进步的基石，共勉！

硬编码
多处定义某个变量，并且变量的值相同：使用含义明显的枚举或常量

如果是新创建逻辑，使用函数式编程或基于接口的编程：可以将通用处理流程抽离出来，而将具体业务逻辑以回调函数的形式传入处理
不要让不同的业务共用相同的函数，每个业务都有各自的函数，并可复用相同的通用逻辑和流程处理
通用逻辑应该确保功能单一
复用而非混杂
直接在原方法里加逻辑
增加了原方法的长度，破坏了可维护性，另一方面，对原方法的既有逻辑造成破坏。
方法：新增一个函数，然后在原方法中调用并说明原因（或者在条件允许的时候反过来，新方法调用原方法）
多业务耦合
先划分业务边界，定义好接口设计和服务依赖关系，再着手开发
代码层次不合理
代码改动逻辑是正确的，然而代码的存放位置不符合当前架构设计约定，导致后续维护成本增加
代码的放置位置不符合当前架构设计约定
代码层次不合理可能导致重复代码
多余或废弃的代码
工程中有一些不用的代码或者是暂时用到的util工具或库函数，或者由于业务变更导致已经废弃不用的代码，或者后来又重写的代码。
后续维护时会造成困惑
方法：确认不用及时删除
使用全局变量
全局变量并没有“错”，但是一旦出现问题，排查和调试问题的时候会比较麻烦
破坏函数的复用性，导致可维护性变差。
代码块越长，全局变量的作用域越广，不易理解和调试
缺乏必要的解释
重要和关键点的代码应该有必要的解释，使用的重要算法缺乏必要的引用出处。对特殊的处理要有必要的说明
原则上，每个方法至少用一个简短的单行注释，适宜地描述方法的用途、业务逻辑、作者及日期。
对于特殊甚至奇葩的需求的特别实现，更要加一些注释，从而后续维护有基础。
较轻微问题
没有技术难度，通过良好的习惯即可避免的问题
命名不贴切
取贴切的名字
声明时未初始化
赋予默认初始值
风格与整体不一致
在工程项目中，遵循工程约定的风格
在个人项目中，可以凸显个性风格
类型转换错误
转换前判断类型
避免类型转换
否定式风格
使用肯定句式，避免双重否定
API参数传递错误
留意顺序正确
参数太多时，使用key-value对象作为options或config传递
单行调用括号过多
在写一个语句内有多个参数多个括号的时候，如果再结合api参数传递错误，IDE没有报错，并且自己也没有意识到错误。
可以拆除一部分变量，并将调用之间的括号用空格隔开，显示出层次感
修改方法签名
不要修改原方法的参数顺序
在最后面增加可选参数
最好的方法是：新建一个方法取复用原方法，然后调用新方法。
打印日志太多
仅保留必需日志
多级数据结构
确保父级结构的数据存在
作用域过大
仅在需要时定义和使用
分支与循环
if-else嵌套条件，需要仔细检查是否符合业务逻辑，如果嵌套太深，是否可以换另一种方式解结
switch语句，case要注意加break，最好加default处理
for循环，要注意结束条件合理避免死循环。对于循环变量的控制，避免出现+1或-1的错误，消除越界错误

复杂隐蔽问题
复杂并发场景下的有一定技术难度的、需要丰富开发与设计经验才能看出来的错误
安全问题
严格检查和屏蔽非法输入
敏感信息加密通信

角色、权限、操作、数据访问分级和控制

随着写的代码量不断的增加，慢慢就接触了封装这个概念。
从一开始的写前后端交互，
当时是web的上机课：
用的是现在已经很少用的jsp。
做的是一个简单的网上商城，用Struts2框架。
（现在想来，貌似了解框架比封装的概念要早很多）
当是是学了用orm来映射数据库后，顺理成章的也写进了框架里。
老师给我们定了3个任务。
1、查询所有的用户
2、查询所有的书并且进行分页
3、查询某位用户买了什么书，并且分页
老师说能做出前两个的，上机课就给优秀，三个都做出来的是满分。
那肯定每个人都是奔着满分去的。
拿到题目后我就开始分析：

第一个就是简单的再Struts2框架里，使用orm关系映射数据库取值。
第二个是在第一个的基础上，加上分页。在那个数据库刚开始会用，sql语句刚熟练的时代，做个分页就觉得很有技术含量了。
第三个比第二个还要难，难在sql语句。要做条件查询。

乍一看，需要写3个方法啊。
但是稍微的写一下这三个方法的时候，就发现，结构很一样。
结构一样，只是传的参数不一样。
所以貌似可以写到一个函数里。然后根据传入参数的不同，用不一样的sql，从而取到对应的值。
虽然当时还不知道封装这个词，但是我有这样一种想法：
把这3个写到一个里面，然后哪里用到了就去调用，外加传参。

当时在写这个函数的时候，写的很简单，类似于以下这种结构：

function fun(num,userid){
数组a，里面装了三条sql；
sql=a[num];
只有在第三条sql语句里，才用得着userid；
sql语句确定了之后，就可以向后端请求数据了。
}
写完后还觉得心里美滋滋，觉得是个很巧妙的想法。

随着后面在学校里还有其他的上机课，
这个函自己又重新写了，让这个函数脱离了那个最原始的情景。可以适配其他的情景。

这可能是我写web端以来第一次那么有意识的去对一个函数做封装。
这个阶段暂且叫做封装思想的萌芽吧。

有了一点点理解我就暂且叫做虾米阶段吧。

等有了更多的经验再来写进阶阶段。

解决方法为：

3、关于marko打包的问题

在marko中打包，
需要关注lasso

https://github.com/lasso-js/lasso#using-lassojs-taglib-with-marko
以上网址为相关的官方的lasso项目，对lasso有详细的解释

在经过学习后，了解到以下的设置：

在server中要设置：
require(‘lasso’).configure({});
*configure里什么也不写表示用marko的默认的设置
在routes文件夹下的每个文件夹下需要新建文件browser.json来标明这个文件夹下的index.marko对应的依赖关系

eg:
在server.js中
require(‘lasso’).configure({
plugins: [
‘lasso-marko’ // Allow Marko templates to be compiled and transported to the browser lasso在marko下的插件
],
outputDir: __dirname + ‘/static’, // Place all generated JS/CSS/etc. files into the “static” dir 打包后的输出路径
bundlingEnabled: isProduction, // Only enable bundling in production 打包环境
minify: isProduction, // Only minify JS and CSS code in production 不保留格式
fingerprintsEnabled: isProduction, // Only add fingerprints to URLs in production
bundles: [
{
“name”: “base”, //打包后的文件名前缀
“dependencies”: [ //列举依赖关系
“./…{文件地址}…/demo1.js”,
“./…{文件地址}…/demo2.js”
]
}
]
});
在routes文件夹下，demo文件夹中
1、新建 browser.json
2、写如下格式的代码：
{
“dependencies”: [
“./…{文件地址}…/demo1.js”,
“./…{文件地址}…/demo2.js”，
“./…{文件地址}…/demo3.js”
]
}

在设置完以上操作后，对项目进行测试。
示例项目中的打包情况如下：
demo文件夹里的browser.json文件里陈述了对应的依赖关系里有demo1.js、demo2.js和demo3.js，会有一个以demo开头的js文件。
在server.js里对demo1.js和demo2.js进行了特殊的设置，把这两个文件打包成一个以base开头的js文件。
在浏览器中查看着两个js文件中的内容，两个文件中的内容并没有相同的部门。
由此可以这么理解，在json文件中，只表明了依赖关系，如果不进行特殊设置的话，会把所有的依赖都打在同一个js文件里，但是如果在server.js进行了特殊的指定，那么被特殊指定的js文件会单独打包。
所有的依赖关系，都会被打包，并且只打包一次。

从promise里面取值

发表于 2018-07-11
字数统计: 630 | 阅读时长 ≈ 2

node中很容易出现异步的问题，为了避免出现钩子陷阱，所以出现了promise这种写法。
从而就会有这样的情景：
在某个方法中，promisify的request连同获取的数据也一并封在了promise对象里。
要想把promise中的数据取出来，我也想了很多办法，像处理平时的对象那样，以把对象转化成字符这种思想为中心，各种方法用过了一遍。
好吧，最后承认promise对象和其他的对象不一样。
只能去网络上搜索相关的方。
首先是这么一种答案：取不出来的。
然后呢？然后就没有然后了。
继续搜，最后终于有了另一种答案：promise里取不出，可以换个方法啊，比如用async/await之类的，反正和promise一样都是消除异步的……这个答案，相当于是曲线救国了。换async/await是可以，但是如果换，就要整个封装的函数进行大改，而且，async/await属于新语法，支不支持还不知道……再结合我要实现的功能，感觉前面还有一些个大坑等着我。
既然写成了promise而且也把request promisify了，硬着头皮，进一步搜索。
果然，搜索引擎是强大的。
找到了一个明白人说的答案，显然是可以把值取出来。
我理解的是这样的：
首先有这样的前提：
promise有三个状态，pending，resolve，reject。每个promise都是从pending到resolve或者reject。当promise到了resolve或者reject状态后，peomise就不再变了。
基于这样的前提下：
那么在promise中的数据也就不会变了。
当return这个promise后，promise的数据也会被return。
好的，下一步就是特别重要的.then()方法
也许，很多人一看，这有什么，我也用过，不就是promise.then()方法吗？
但是这个.then()方法值得我们好好讨论一下
——此处占位写.then()的相关资料——-
在看了很多的资料后，有一个特别通俗的说法：这个.then()方法可以认为是一个callback()，后面可以搭配.catch()可以捕捉来自promise里的错误，不用设置reject，就可以把错误捕捉到。
eg
在eg.js中
module.exports = {
demo:function (){
var promise = new promise((resolve,reject)=>{
//写个取值的操作
});
return promise;
}
}

在另一个文件中调用的时候：
var eg = require(‘/eg.js’);
var p = eg.demo;
p.then(function(data)){
console.log(data);
//在此处可以写对数据的操作
}

字数统计: 1,542 | 阅读时长 ≈ 7

基础是进步的基石。

JS和HTML的交互基于事件。

事件流
描述的是从页面中接受事件的顺序
IE：事件冒泡流
Netscape：事件捕获流

事件冒泡：即事件最开始由最具体的元素（文档中嵌套层次最深的那个节点）接收，逐级向上传播至最不具体的那个节点（文档）。
click在input->div->body->document

事件捕获：不太具体的节点应该更早接收到事件， 而最具体的节点最后接收到事件。
document->body->div->input
使用事件处理程序
1、HTML事件处理程序
把事件直接加在HTML结构里。
缺点：HTML和JS代码紧密耦合
2、DOM0级事件处理程序
把一个函数赋值给一个事件的处理程序属性，是用的比较多的方法
简单，跨浏览器
先取一个元素，把事件作为元素的属性，没有HTML事件的缺点。
var btn = document.getElementById(‘dd);
btn.onclick=function(){}
btn.onclick=null;//取消了onclick事件
3、DOM2级事件处理程序
定义了两个方法：
用于处理指定和删除事件处理程序的操作
addEventListener()和removeEventListener()
接收3个参数：
要处理的事件名、作为事件处理程序的函数和布尔值（true：事件捕获，false：事件冒泡）
var btn = document.getElementById(‘dd);
btn.addEventListener(‘click’,fun,false); //onclick换成click，onmouseover换成mouseover,以此类比，设置成false作为一个冒泡事件从而让大多数浏览器兼容
btn.removeEventListener(‘click’,fun,false) //传递和addEventListenery一样的参数才能删除，而且addEventListener的事件必须由removeEventListener才能删除
DOM0级和DOM2级可以添加多个事件以及多个事件处理程序
但是IE不支持，因此以下是IE的事件处理程序
4、IE事件处理程序
两个参数，事件处理程序的名称和事件处理程序的函数
attachEvent()添加事件
detachEvent()删除事件
不使用第三个参数的原因是：IE8以及以前的版本只支持事件冒泡。
var btn = document.getElementById(‘dd);
btn.attachEvent(‘onclick’,fun); //这个时候的事件又要加on了，后面第二个参数是函数名
btn.detachEvent(‘onclick’,fun) //删除事件

除了鼠标事件之外，还有键盘事件

keyDown:当用户按下键盘上的任意键时触发，而且如果按住不放的话，会重复触发此事件
keyPress:当用户按下键盘上的字符键时触发，而且如果按住不放的话，会重复触发此事件
keyUp:当用户释放键盘上的键时触发，按下然后抬起后触发事件

event对象的keyCode属性用于得到键盘对应键的键码值
